# AI Context Bridge — Claude Working Notes

VS Code extension. Shared "blackboard" memory + skill control center for multi-model AI workflows. State-management only — no token math, no API calls.

## What this codebase is

A VS Code extension that gives multiple AI agents (Claude, GPT, Gemini, etc.) a shared on-disk memory so they can hand off work without re-explaining context. The user controls which AI capabilities ("skills") are live, frozen, or gated by a confirmation prompt.

Core value: **visibility + control**, not orchestration. We don't call models. We persist what they thought, what files they pinned, what tools they're allowed to use.

## Architecture in 30 seconds

```
.aicb/
├─ state.json       ← single source of truth (PersistedState)
└─ snapshots.json   ← rollback points

src/
├─ memory/                ← Phase 1 (T1.1, T1.2)
│  ├─ types.ts            JSON schema
│  └─ MemoryManager.ts    atomic write queue, EventEmitter
├─ views/                 ← Phase 2 (T2.1–T2.3)
│  ├─ SkillTreeProvider           sidebar — 3-state toggle
│  ├─ PinnedFilesProvider         sidebar — pinned files
│  ├─ SnapshotProvider            sidebar — snapshots
│  ├─ ThoughtTimelineView         webview — model badges + handoff hints
│  └─ PinnedDecorationProvider    📌 badge on file tabs
├─ safety/                ← Phase 3 (T3.1–T3.2; T3.3 lives in MemoryManager)
│  ├─ ToolFilter         filter tool[] by ENABLED/ASK/DISABLED
│  └─ HITLManager        modal: allow once / allow session / deny
└─ extension.ts          activation, command wiring, status bar
```

`MemoryManager` is the only writer. Everything else subscribes to `onDidChange` and re-renders.

## Conventions to follow

- **State changes go through MemoryManager.** No direct fs writes. The `writeQueue` serializes writes and does atomic rename (`.tmp` → real path) so partial writes can't corrupt `state.json`.
- **Auto-sync respects `aiContextBridge.autoSync` setting.** If false, mutations only fire `onDidChange`; user must run `Force Sync` (status bar click) to persist.
- **Kill switch is sticky.** Stored in workspace settings (`aiContextBridge.killSwitchEngaged`), mirrored into state on activation. When ON, `effectiveStatus()` returns `DISABLED` for every skill.
- **Skill IDs are stable identifiers** (`shell.exec`, `fs.write`). Display names are mutable; IDs are not.
- **Webview CSP is strict.** Only `webview.cspSource` + inline styles/scripts allowed. No external resources.
- **No fill colors in activity-icon.svg.** Uses `stroke="currentColor"` so VS Code themes the icon. Hard-coded colors break light theme.

## Public API (for other extensions)

```ts
const ext = vscode.extensions.getExtension("jeurboy.ai-context-bridge");
const api = await ext?.activate(); // { memory, toolFilter, hitl }

api.memory.addThought({ modelId: "claude-opus-4-7", text: "..." });
api.memory.pinFile({ path, pinnedBy: "gpt-5" });
const { enabled, ask, disabled } = api.toolFilter.filter(myTools);
const decision = await api.hitl.authorize({
  skillId,
  skillName,
  modelId,
  summary,
});
```

## Things that tripped me up before

- **`viewsContainers.activitybar[].icon` must be a file path**, not `$(codicon)`. Codicon refs work for `commands` and `views[].icon` but NOT activity bar containers.
- **`structuredClone` requires Node 17+.** package.json doesn't pin engines.node; VS Code 1.85+ ships Node 18 so it's fine, but if we ever target older we need a fallback.
- **TreeItem `command.arguments`** receive the _node_ the user clicked, not just the id. The command handlers use `resolveSkillId()` to accept either string id or node-with-skill — keep that pattern when adding new TreeItem commands.
- **FileDecorationProvider must filter by `uri.scheme === 'file'`.** Decoration is called for git, search, output channels — anything that isn't a real file will throw downstream.

## What's NOT in scope

- Calling AI APIs / token counting / streaming
- Network sync between machines (everything is local-only by design — privacy)
- Auto-detecting which AI is currently active
- Rewriting tool calls — `ToolFilter` returns the partition; the _consumer_ decides what to do with it
